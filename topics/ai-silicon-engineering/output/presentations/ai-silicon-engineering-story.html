Chapter 1

What is a chip, really?

Start with the plainest idea: a chip is a tiny physical object that can turn signals on and off very quickly. Everything else in this topic grows from that one fact.

[[semiconductor-fundamentals]]

Four words to know first

These are the minimum terms you need. No jargon required beyond this point.

Transistor

A tiny switch. It can let electricity pass or block it.

Chip

A piece of silicon containing billions of those tiny switches wired together.

Wafer

A round disk of silicon that holds many chips before they are cut apart.

Package

The outer shell that protects the chip and connects it to the rest of the computer.

You can ignore labels like PMOS and CMOS for now. The important idea is simply that chips are built from many controlled switches, and switching them efficiently is what makes computation possible.

The smallest possible chip idea

This picture is just: input goes in, tiny switch does its job, output comes out. That is the basic pattern behind all digital hardware.

Chapter 2

Different workloads want different machines.

CPU, GPU, TPU, NPU, FPGA, and ASIC are not six names for the same thing. They are six answers to the same tradeoff: flexibility versus efficiency.

[[processing-unit-types]]

The decision axis

Move right for specialization, move up for raw flexibility. No design gets everything.

Flexibility
Efficiency
CPU general purpose, low-latency control
GPU programmable parallelism, mature software
TPU matrix multiply first, compiler driven
NPU edge inference, low power
FPGA reconfigurable bridge to hardware reality
ASIC maximum specialization, minimum waste

What the story tells you

The same model can be a CPU task, a GPU task, a TPU task, or a custom accelerator task depending on volume, stability, latency, and software support.

GPU wins when the workload changes CUDA, TensorRT, and the rest of the software stack make GPUs the safe choice for unstable or experimental model shapes.
TPU wins when the workload is regular A compiler-friendly graph plus a systolic array can extract far more useful work from each watt.
FPGA wins when you need a bridge It is not the fastest path to scale, but it is a crucial proving ground between code and silicon.
ASIC wins when the problem is fixed If the workload is stable enough, the cost of specializing the hardware can pay back at volume.
The important lesson is not which chip is fastest in a vacuum. It is which chip can actually be deployed for the workload you have.
Chapter 3

AI speed comes from moving data less.

Accelerators are not just math engines. They are memory and dataflow machines. The question is how to keep compute fed without wasting bandwidth.

[[ai-accelerator-design]]

The roofline view

A workload is either compute-bound or memory-bound. Most optimization is really about raising operational intensity so useful math stays ahead of data movement.

The dataflow patterns

The hardware philosophy is baked into how weights, activations, and partial sums move through the compute fabric.

Weight-stationary

Keep weights resident, stream activations through. Great when the same parameters are reused across many inputs.

Output-stationary

Keep partial sums local and reduce writeback pressure. Good when the output accumulation path is the expensive part.

Row-stationary

Reuse weights, activations, and partial sums together. This is where energy efficiency starts to look like a scheduling problem.

Input-stationary

Hold activations still and let weights flow. That makes sense when input reuse is the dominant opportunity.

The real prize is reuse. If you can keep data on-chip longer, the accelerator looks much faster even before you touch raw FLOPS.
Chapter 4

The compiler is what makes specialization usable.

A rigid accelerator is dead weight unless software can map models onto it. Compilers are the bridge from high-level intent to machine-specific execution.

[[ai-compilers]]

Lowering path

The same computation can be lowered to CUDA, TPU instructions, LLVM IR, or a custom backend. The model stays the same; the execution path changes completely.

Why this matters

Compilers are not a convenience layer. They are the mechanism that translates one mathematical graph into a machine-specific plan that can actually run well.

Fusion

Combine small ops so memory bandwidth is not wasted on intermediate writes.

Tiling

Break tensors into chunks that fit local memory and match the machine's execution shape.

Scheduling

Pick the loop order, memory placement, and kernel shape that best match the hardware.

Backend generation

Emit the actual instruction stream for the target platform.

Without a compiler stack, a custom accelerator is only a lab artifact. With a compiler stack, it becomes a product.
Chapter 5

Packaging is now part of the architecture.

The package determines memory bandwidth, thermal headroom, yield, and how far a design can scale. In AI silicon, package design moves upstream into the main architectural choice.

[[advanced-packaging]]

Stacked system view

Think of the module as a system of layers: compute die, interconnect, HBM, substrate, and board all constrain one another.

Why the package decides the winner

More compute is pointless if the memory cannot feed it. More HBM is pointless if the package cannot place and cool it. More die area is pointless if yield breaks.

Bandwidth

Package choice sets how much memory can sit close enough to matter.

Thermals

High-performance AI modules are thermal systems as much as they are compute systems.

Economics

Substrate cost, assembly complexity, and capacity bottlenecks shape what ships at scale.

CoWoS, CoPoS, and CoWoP are not just packaging abbreviations. They are attempts to make the memory and compute system physically plausible at the size AI now demands.
Chapter 6

Testability and yield decide whether the chip survives contact with reality.

Design for Test makes the invisible visible. Yield turns a prototype into a manufacturable product. Without them, the best architecture still fails economically.

[[design-for-test]] + [[vlsi-design-flow]]

The closing loop

DFT, scan, ATPG, and physical verification convert a huge die from a speculative object into a shippable one.

Scan

Expose internal state so stuck-at faults and hidden logic issues can be detected cheaply.

ATPG

Generate test patterns that probe defects before bad silicon reaches customers.

Yield

Every additional defect opportunity changes the economic shape of the chip.

Signoff

DRC, LVS, STA, and physical verification decide whether the design is real.

Visual test loop

The feedback loop is simple: manufacture, test, bin, learn, and iterate.
