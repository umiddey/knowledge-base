# finAPI Access User

The Access User is a synthetic user identity in finAPI's system, created programmatically by your application and used as the "subject" of all Web Form and payment operations. It is NOT an end-user account — it's a service account your backend creates, stores, and reuses. The access user's credentials (id + password) enable the user-scoped OAuth token (`grant_type=password`), which is required for all bank-related API calls including Web Form creation and payment status queries.

The architectural pattern: one Access User per tenant (SaaS client). You create it once when the tenant configures their finAPI integration, store the generated id + password encrypted alongside the client credentials, and reuse it for all that tenant's payment operations. This isolates payment context cleanly between tenants. The alternative — deleting the user after each payment — is valid for one-time use cases but adds latency (user creation + deletion per transaction) and doesn't work well for polling-based flows.

## Key Points

- **Created via**: `POST /api/v1/users` with a client access token. You supply the id, password, and optionally email/phone.
- **id format**: Any string — in Aetherios: `aetherios-{uuid_hex[:16]}` for readability and uniqueness.
- **password**: Random, generated by your application. finAPI returns it in plaintext in the creation response. Store it encrypted immediately.
- **`isAutoUpdateEnabled: false`**: Always set this for service accounts — you don't want automatic bank data refresh jobs running on a synthetic user.
- **Reuse vs delete**: For recurring payment use cases, reuse the same Access User. `DELETE /users` is optional, only needed if you want to clear user data from finAPI's system (privacy compliance, license headcount).
- **Language preference**: Web Forms rendered for this user default to the browser's language. If the access user's preferences are stored across sessions (they are, unless the user is deleted), language choice persists. For service accounts where you control the browser context, this is usually irrelevant.
- **Provisioning timing**: Should happen at finAPI config save time (when the admin enters finAPI credentials in settings), not at payment time. This avoids latency on the payment initiation path.

## Example

Access User provisioning in Aetherios at finAPI configuration save time:

```python
# In PaymentProcessorConfigService.create_or_update_finapi_config()
# For "real" mode (not mock):

service = FinapiService(client_id=..., client_secret=..., api_base_url=..., mock_mode=False)
user_creds = await service.provision_access_user()
# → { "access_user_id": "aetherios-a3f92b1c4d5e6f78", "access_user_password": "Xk9mP2qR7vN4jL8w" }

# These get stored in payment_processor_config.encrypted_credentials alongside:
# client_id, client_secret, api_base_url, sender_iban, etc.

# At payment time, FinapiService is initialized with:
return FinapiService(
    client_id=decrypted.get("client_id"),
    client_secret=decrypted.get("client_secret"),
    access_user_id=decrypted.get("access_user_id"),      # ← provisioned user
    access_user_password=decrypted.get("access_user_password"),
    ...
)
```

The finAPI Access User is analogous to a "service account" in cloud IAM — it exists purely for programmatic access, has no human behind it, and its credentials are managed by the application.
