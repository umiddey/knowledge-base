# finAPI OAuth2 Token Model

finAPI uses OAuth 2.0 with two distinct actor types — clients and users — each requiring their own access token. Every API call must carry a valid token. Understanding which token is needed for which operation is the first thing that trips up integrators: using a client token where a user token is required silently fails or returns 401.

A **client** represents your application (the SaaS company, the fintech). A **user** represents an end-user within your application — in finAPI's model, this is a synthetic identity created specifically for Web Form operations, not a real person's login. The hierarchy is: your client credentials authorize creation of users; user credentials authorize payment operations. Both token types expire in ~3600 seconds (1 hour), with a recommended 60s safety margin before refresh.

The flow is always: `client_credentials` → `POST /api/v1/users` → `password` grant → operations. For most Web Form payment flows, you provision an Access User once at finAPI config setup time, store their id + password encrypted, and reuse that user for all subsequent payment operations for that tenant. Deleting users after one-time operations is optional — only relevant if you don't want to accumulate user accounts or have privacy concerns.

## Key Points

- **Client token** (`grant_type=client_credentials`): Used for admin operations — creating users, editing client configuration. Does NOT authorize bank-related operations.
- **User token** (`grant_type=password`): Used for all bank-related operations — creating web forms, checking web form status, fetching payment status. Requires `client_id`, `client_secret`, `username`, `password`.
- **Token TTL**: Both expire in 3599 seconds. Cache with a 60s safety margin. On expiry, re-request with the same credentials.
- **Sandbox vs Live**: Token endpoints differ by base URL (`sandbox.finapi.io` vs `live.finapi.io`) but the OAuth flow is identical.
- **Security**: Never include an access token in a Web Form URL query string (it would be logged, cached, visible in redirects). Use callback/redirect URLs with opaque identifiers instead.
- **Access User design**: finAPI's Access User is a synthetic service account, not an end-user identity. Its credentials (id, password) are generated by you, stored encrypted, and used programmatically — the end-user never knows this user exists.

## Example

Full token sequence for a finAPI payment initiation:

```python
# Step 1: Client token
POST https://sandbox.finapi.io/oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials&client_id=MY_APP&client_secret=MY_SECRET
# → { "access_token": "fnsX...", "expires_in": 3599 }

# Step 2: Create Access User (one-time, at config setup)
POST https://sandbox.finapi.io/api/v1/users
Authorization: Bearer fnsX...
{ "id": "aetherios-a3f92b1c", "password": "Xk9mP2qR7vN4jL8w" }
# → { "id": "aetherios-a3f92b1c", "password": "Xk9mP2qR7vN4jL8w" }
# Store id+password encrypted in payment_processor_config

# Step 3: User token (every payment session)
POST https://sandbox.finapi.io/oauth/token
grant_type=password&client_id=MY_APP&client_secret=MY_SECRET
&username=aetherios-a3f92b1c&password=Xk9mP2qR7vN4jL8w
# → { "access_token": "nnsX...", "expires_in": 3599 }

# Step 4: All Web Form and payment operations use the user token
POST https://webform-sandbox.finapi.io/api/webForms/standalonePayment
Authorization: Bearer nnsX...
```

In Aetherios, `FinapiService._ensure_client_access_token()` and `_ensure_user_access_token()` handle caching and refresh transparently. Token cache is in-memory on the `FinapiService` instance.
